# Aula 02 — O que é um sistema

## Objetivo da aula

O objetivo desta aula é construir uma compreensão inicial, mas consistente, do conceito de sistema. Ao final, o aluno deve conseguir explicar que um sistema não é apenas uma coisa, uma tela, um programa ou uma ferramenta. Um sistema é um conjunto de componentes relacionados, organizado para cumprir uma finalidade, com limites, entradas, transformações, saídas, interações e dependências.

Esta aula dá continuidade direta à Aula 01. Depois de entender tecnologia da informação como relação entre pessoas, processos, dados e tecnologia, o próximo passo é entender como essas partes podem se organizar. A palavra "sistema" aparecerá durante toda a formação: sistema de informação, sistema computacional, sistema corporativo, sistema operacional, sistema legado, sistema crítico, sistema batch, sistema online. Se o aluno não compreender a ideia geral de sistema agora, poderá confundir conceitos futuros e tratar tudo como se fosse apenas código ou tela.

## O problema que esta aula resolve

Muitos iniciantes usam a palavra "sistema" para quase qualquer coisa em tecnologia. Uma tela vira sistema. Um aplicativo vira sistema. Um programa isolado vira sistema. Uma ferramenta vira sistema. Às vezes, a palavra é usada apenas como sinônimo de "software", sem considerar usuários, dados, regras, processos, infraestrutura, permissões, integrações e operação.

Essa confusão parece pequena no início, mas causa problemas de compreensão. Se alguém acredita que sistema é apenas a tela que aparece para o usuário, não percebe os dados que sustentam aquela tela. Se acredita que sistema é apenas código, ignora processos de negócio, regras, entradas, saídas, logs e responsabilidades humanas. Se acredita que uma ferramenta sozinha é o sistema inteiro, pode imaginar que trocar uma ferramenta resolve automaticamente um problema que, na verdade, envolve processo mal definido, dado incorreto ou integração frágil.

Esta aula resolve essa lacuna ao apresentar sistema como conceito estrutural. Um sistema envolve partes, mas não é só uma soma de partes. Envolve componentes, mas também relações. Envolve funcionamento, mas também propósito. Envolve limites, mas também comunicação com o ambiente externo. A partir dessa ideia, será possível entender melhor sistemas computacionais e corporativos sem reduzir tudo a programa ou aplicativo.

## Introdução

Na Aula 01, tecnologia da informação foi apresentada como uso organizado de recursos técnicos, pessoas, processos e métodos para lidar com informações. Essa definição já sugeria uma ideia importante: a tecnologia não aparece isolada. Ela se organiza em relações. Uma pessoa informa dados, um processo define o que deve acontecer, uma regra orienta decisões, uma ferramenta apoia uma etapa, um sistema registra e consulta informações, uma saída permite agir.

O conceito de sistema ajuda a enxergar essa organização. Quando olhamos apenas para uma parte, perdemos o comportamento do conjunto. Uma empresa não é apenas seu prédio, seus computadores ou seus funcionários vistos separadamente. Um banco não é apenas o aplicativo que o cliente usa. Um processo de pagamento não é apenas o botão "confirmar". Um sistema de folha de pagamento não é apenas o programa que calcula valores. Cada um desses exemplos envolve componentes relacionados que atuam com alguma finalidade.

Estudar sistemas antes de programação é importante porque programas costumam fazer parte de sistemas maiores. Um programa pode validar um cadastro, calcular um valor, ler um arquivo, gerar um relatório ou atualizar um registro. Mas, em ambiente real, ele raramente existe sozinho. Ele recebe dados de algum lugar, depende de regras, produz saídas, pode ser chamado por outro sistema, pode falhar por causa de um arquivo ausente, pode afetar uma operação de negócio e pode exigir rastreabilidade.

Portanto, entender sistema é aprender a olhar para relações. Essa visão será necessária para compreender hardware, software e pessoas na Aula 03; dados, informação e conhecimento na Aula 04; processamento nas Aulas 05 e 06; programas, código e execução nas Aulas 07 a 09; arquivos, ambientes e erros nas aulas seguintes.

## Sistema como conjunto organizado

Um sistema pode ser entendido como um conjunto de componentes relacionados que operam de forma organizada para cumprir uma finalidade. Essa definição tem três ideias centrais: componentes, relações e finalidade. Se uma dessas ideias desaparece, o conceito fica incompleto.

Um conjunto qualquer de partes não é necessariamente um sistema. Peças de uma máquina espalhadas sobre uma mesa formam um conjunto de partes, mas não estão organizadas para funcionar. Documentos acumulados em uma gaveta formam um conjunto de papéis, mas não necessariamente um sistema de informação. Computadores comprados por uma empresa formam um conjunto de equipamentos, mas não resolvem um processo se não houver software, dados, regras, usuários e forma de uso.

Estrutura também não é, sozinha, sinônimo de sistema. Uma estrutura indica organização interna: partes posicionadas, categorias definidas, relações possíveis. Mas um sistema envolve comportamento. Ele recebe estímulos ou entradas, realiza alguma transformação, produz resultados e interage com um ambiente. Um arquivo organizado pode fazer parte de um sistema; uma tabela pode fazer parte de um sistema; uma hierarquia de departamentos pode fazer parte de um sistema. O sistema aparece quando essas estruturas atuam em relação a uma finalidade.

Por exemplo, uma pilha de formulários vazios é apenas um conjunto de documentos. Quando esses formulários são usados em um processo de cadastro, preenchidos por usuários, conferidos por atendentes, registrados em uma base de dados, consultados por outra área e usados para tomada de decisão, eles passam a participar de um sistema mais amplo. O valor do sistema não está no papel isolado, mas na organização das partes em torno de uma finalidade.

Essa distinção é fundamental para tecnologia da informação. Um computador, uma planilha, um formulário, um banco de dados ou um programa pode ser um componente. O sistema é o arranjo organizado desses componentes com um propósito. Por isso, a pergunta "qual é o sistema?" não deve ser respondida apenas apontando para uma tela. É preciso entender quais partes participam, como se relacionam e que resultado precisam produzir.

## Componentes, relações e propósito

Componentes são partes identificáveis de um sistema. Em um sistema técnico, um componente pode ser um equipamento, um programa, um arquivo, uma conexão, uma base de dados ou uma tela. Em um sistema organizacional, pode ser uma área da empresa, uma pessoa, um documento, uma regra, uma etapa de aprovação ou um canal de atendimento. O tipo de componente depende do sistema observado.

As relações entre componentes são tão importantes quanto os componentes em si. Dois sistemas podem usar componentes parecidos e produzir comportamentos muito diferentes se as relações entre eles forem diferentes. Uma empresa pode ter atendentes, cadastro, sistema de consulta e área de análise. Se essas partes não trocam informações corretamente, o atendimento falha. Outra empresa com componentes semelhantes pode funcionar melhor porque define claramente quem registra, quem confere, quem aprova e como a informação circula.

O propósito orienta a existência do sistema. Um sistema de cobrança existe para controlar valores a receber, vencimentos, pagamentos, juros, notificações e histórico. Um sistema de cadastro existe para registrar e manter dados de entidades relevantes, como clientes, produtos ou beneficiários. Um sistema de pagamento de salários existe para transformar dados funcionais, eventos, descontos e regras em valores de pagamento e registros legais. Sem propósito, não há critério para dizer se o sistema funciona bem.

Alterar um componente pode afetar o sistema inteiro. Se uma regra de validação muda, dados antes aceitos podem passar a ser recusados. Se um campo do cadastro muda de significado, relatórios podem ser interpretados incorretamente. Se uma integração deixa de enviar informações no horário esperado, uma etapa posterior pode atrasar. Se uma pessoa executa uma etapa fora do procedimento, dados podem ser registrados de forma incompleta.

Essa ideia prepara o aluno para uma postura importante: em sistemas, mudanças precisam ser pensadas por impacto. Não basta perguntar "o componente foi alterado?". É preciso perguntar "quem depende dele?", "que dados ele recebe?", "que saídas produz?", "que regras aplica?", "que processos serão afetados?", "que comportamento do todo pode mudar?".

## Entrada, processamento, saída e retroalimentação

Muitos sistemas podem ser observados por um modelo inicial: entrada, processamento, saída e retroalimentação. Esse modelo será aprofundado em aulas futuras, especialmente na Aula 06, mas já é útil para compreender que sistemas interagem com algo, transformam algo e produzem algum resultado.

Entrada é aquilo que o sistema recebe. Pode ser uma solicitação, um dado digitado, um arquivo, uma mensagem, um documento, um pedido, uma leitura de sensor ou uma autorização. Em um processo de abertura de conta, a entrada pode incluir dados pessoais, documentos, comprovantes e aceite de termos. Em um processo de cobrança, a entrada pode incluir contrato, vencimento, valor, situação de pagamento e dados de contato.

Processamento é a transformação realizada pelo sistema. Pode envolver conferência, cálculo, validação, classificação, consulta, aprovação, recusa, registro ou comunicação com outro sistema. Processamento não precisa ser entendido ainda como algoritmo formal. Neste momento, basta compreender que o sistema faz algo com a entrada de acordo com regras e organização internas.

Saída é o resultado produzido. Pode ser uma resposta na tela, um relatório, uma atualização de cadastro, um boleto emitido, uma notificação enviada, uma autorização concedida, um pagamento efetivado ou um registro de log. Saída não é apenas aquilo que aparece visualmente ao usuário; muitas saídas são arquivos, mensagens, atualizações internas ou evidências para auditoria.

Retroalimentação, ou feedback, ocorre quando uma saída ou resultado retorna como informação para ajustar o funcionamento do sistema. Em um atendimento ao cliente, reclamações recorrentes podem indicar que uma etapa do processo está falhando. Em um sistema de cobrança, pagamentos não identificados podem levar à revisão de dados bancários ou regras de conciliação. Em um processo de análise de pedidos, recusas frequentes por documentação incompleta podem indicar que o formulário inicial precisa ser melhorado.

Esse modelo ajuda a evitar uma visão estática. Sistemas não são apenas estruturas paradas; eles recebem, transformam, produzem e aprendem ou se ajustam a partir de resultados observados. Em sistemas corporativos, esse movimento precisa ser controlado porque entradas incorretas, processamentos mal definidos ou saídas ambíguas podem gerar impacto operacional.

## Fronteira, ambiente e interface

A fronteira de um sistema é o limite conceitual que separa o que será considerado dentro do sistema e o que será considerado fora dele. Essa fronteira não precisa ser física. Ela depende do que estamos analisando. Se estamos analisando um sistema de cadastro, talvez a tela, as regras de validação, a base de dados e os usuários internos estejam dentro do recorte. Clientes, sistemas externos e documentos recebidos podem estar no ambiente externo, mesmo interagindo com o sistema.

Definir fronteira é importante porque nenhum sistema real pode ser analisado como se incluísse o mundo inteiro. Para entender um problema, precisamos delimitar o que será observado. Se uma rotina de emissão de boletos falha, podemos começar perguntando se o problema está dentro do sistema de cobrança, na integração com o banco, nos dados do cliente, na regra de vencimento, no arquivo enviado ou no procedimento operacional. A fronteira ajuda a organizar a investigação.

O ambiente externo é tudo aquilo que está fora da fronteira do sistema, mas pode influenciar seu funcionamento. Um sistema de vendas depende de clientes, fornecedores, meios de pagamento, regras fiscais, estoque, transportadoras e canais de atendimento. Nem todos esses elementos estão dentro do sistema de vendas, mas todos podem afetá-lo.

Interface é o ponto de contato entre sistemas, componentes ou pessoas. Uma tela é uma interface entre usuário e sistema. Um arquivo pode ser interface entre dois sistemas. Uma mensagem enviada por integração pode ser interface entre aplicações. Um formulário pode ser interface entre uma pessoa e um processo administrativo. Interfaces são importantes porque definem como a comunicação acontece: que dados entram, em que formato, em que momento, com quais regras e com que resposta esperada.

Essa ideia será importante futuramente para integração entre sistemas. Em ambientes corporativos, sistemas raramente trabalham isolados. Um sistema de vendas pode conversar com estoque, financeiro, emissão fiscal, logística e atendimento. Um erro na interface entre dois sistemas pode produzir uma falha que parece estar em uma tela, mas cuja causa está em dados enviados incorretamente, formato incompatível, permissão ausente ou regra de integração mal definida.

## Tipos de sistemas

Um sistema natural existe na natureza, sem ter sido projetado diretamente por uma organização humana para um fim técnico. O corpo humano, um ecossistema, o clima e o ciclo da água podem ser observados como sistemas porque possuem componentes, relações, entradas, saídas e equilíbrio dinâmico. Essa ideia mostra que sistema é um conceito mais amplo que computação.

Um sistema social envolve pessoas, papéis, regras, comunicação e formas de organização. Uma escola, uma equipe de trabalho ou uma instituição podem ser observadas assim, porque o comportamento do conjunto depende das relações entre pessoas, expectativas e decisões.

Um sistema técnico é construído por pessoas para cumprir alguma função prática usando artefatos, ferramentas, máquinas ou métodos. Um sistema de transporte, uma rede elétrica e uma linha de produção são exemplos de componentes projetados para interagir de forma controlada.

Um sistema organizacional reúne estruturas, pessoas, processos, regras e recursos que permitem a uma organização funcionar. Uma empresa vista dessa forma inclui departamentos, responsabilidades, fluxos de trabalho, decisões, documentos, metas e controles. Tecnologia pode apoiar esse sistema, mas não o substitui integralmente.

Um sistema de informação coleta, organiza, processa, armazena, transmite e apresenta informações para apoiar operações, decisões, controles e prestação de contas. Ele pode envolver tecnologia digital, mas também envolve processos e pessoas. Um sistema de atendimento, por exemplo, inclui dados registrados, regras de classificação, histórico, consultas, relatórios e uso pelas equipes.

Um sistema computacional é aquele em que recursos computacionais participam diretamente do funcionamento. Ele pode incluir hardware, software, dados, redes, usuários, permissões, arquivos, configurações e ambiente de execução. A Aula 03 começará a detalhar parte desses componentes.

Um sistema corporativo sustenta processos importantes de uma organização. Ele costuma lidar com dados relevantes, usuários diversos, regras de negócio, controles, histórico, segurança, integração e manutenção. Sistemas bancários, fiscais, previdenciários, contábeis, de cobrança, folha de pagamento, estoque e atendimento entram nessa categoria.

Esses tipos não são caixas totalmente separadas. Um sistema corporativo pode ser um sistema de informação e também um sistema computacional. Ele existe dentro de um sistema organizacional e apoia processos sociais e técnicos. A distinção serve para enriquecer a análise, não para criar classificações rígidas.

## Sistema não é apenas software

Um erro comum é reduzir sistema a software. Software é parte importante de muitos sistemas, mas não é o sistema inteiro. Um sistema computacional real pode envolver programas, dados, infraestrutura, regras, usuários, permissões, integrações, logs, segurança, manutenção, operação e documentação.

Uma tela de cadastro, por exemplo, não é o sistema completo. Ela é uma interface. Por trás dela, pode haver validações, armazenamento de registros, controle de acesso, regras sobre campos obrigatórios, integração com outros cadastros, logs de alteração, relatórios e procedimentos de correção. Se a tela funcionar, mas os dados forem gravados de forma incorreta, o sistema não está funcionando adequadamente como conjunto.

Um aplicativo também não deve ser confundido com todo o sistema. O aplicativo pode ser o canal de acesso do usuário, enquanto outros componentes executam validações, consultam bases, registram eventos, comunicam-se com serviços externos e produzem saídas. O usuário toca em uma opção; o sistema como todo coordena várias partes.

Uma ferramenta sozinha também não resolve necessariamente um problema. Uma ferramenta de relatórios pode gerar gráficos, mas se os dados de origem forem ruins, o relatório será enganoso. Uma ferramenta de atendimento pode organizar chamados, mas se o processo for mal definido, o sistema continuará frágil.

## Empresas como sistemas

Uma empresa pode ser observada como um sistema porque possui componentes relacionados que atuam para cumprir objetivos. Esses componentes incluem departamentos, pessoas, processos, documentos, sistemas de informação, clientes, fornecedores, regras internas, obrigações legais, recursos financeiros e metas operacionais. O funcionamento da empresa não está em uma única parte, mas na coordenação entre elas.

Em um banco, por exemplo, há clientes, contas, canais de atendimento, sistemas de transação, áreas de risco, regras regulatórias, sistemas antifraude, operações de pagamento, auditoria, infraestrutura tecnológica e equipes de suporte. Uma transferência bancária aparentemente simples depende de identificação do cliente, autorização, saldo, regras de limite, registro, comunicação entre sistemas e confirmação. Se um componente falha, o impacto pode atingir a operação inteira.

Em um órgão público, um processo de concessão de benefício envolve atendimento, documentação, análise de critérios, cadastro, decisão, pagamento, recurso, fiscalização e prestação de contas. O sistema organizacional precisa lidar com direitos, regras legais, orçamento, prazos e histórico. A tecnologia da informação apoia esse sistema, mas o processo envolve também pessoas, normas e responsabilidades.

## Processos de negócio como sistemas

Um processo de negócio também pode ser analisado como sistema. Ele possui entradas, etapas, regras, responsáveis, decisões, saídas e formas de controle. Observar processos como sistemas ajuda a identificar dependências e consequências.

Na abertura de conta, a entrada inclui dados pessoais, documentos, aceite de condições e, em alguns casos, análise de risco. O processamento envolve conferência de identidade, validações cadastrais, aplicação de regras, aprovação ou recusa e criação de registros. A saída pode ser conta aberta, solicitação pendente, recusa justificada ou necessidade de documentação adicional. Se uma validação falha, uma conta pode ser aberta com dados incorretos ou uma pessoa legítima pode ser recusada indevidamente.

No pagamento de salário, as entradas incluem cadastro de funcionários, jornada, faltas, horas extras, benefícios, descontos, impostos e regras trabalhistas. O processamento calcula valores, aplica regras, gera registros e prepara pagamentos. As saídas incluem valores a pagar, comprovantes, relatórios, obrigações legais e registros contábeis. Uma falha em um campo, como cargo, evento ou desconto, pode afetar salário, imposto e histórico.

Na concessão de benefício público, as entradas incluem dados do cidadão, documentos, critérios legais, histórico e solicitações. O processamento envolve conferência, análise de elegibilidade, decisão, registro e eventual pagamento. As saídas incluem concessão, indeferimento, exigência de documentos, comunicação ao cidadão e registros para auditoria. Esse processo mostra claramente como sistema não é apenas software: há lei, atendimento, documentação, análise humana, dados, controles e tecnologia.

Esses exemplos mostram que processos possuem comportamento sistêmico. Uma etapa depende de outra. Uma entrada incorreta pode produzir saída incorreta. Uma decisão sem registro prejudica auditoria. Uma interface mal definida entre áreas cria atraso. Pensar em sistema ajuda a enxergar essas relações.

## Sistemas de informação

Um sistema de informação é um sistema que coleta, organiza, processa, armazena, transmite e apresenta informações para apoiar operações, decisões, controles e prestação de contas. Ele existe porque organizações precisam transformar dados dispersos em informação útil e confiável.

Esse conceito se conecta diretamente à Aula 01. Na aula anterior, vimos que tecnologia da informação não é apenas computador; ela existe para lidar com informação dentro de contextos humanos e organizacionais. Agora, o sistema de informação aparece como uma forma organizada de realizar isso. Ele define como dados entram, como são tratados, onde são armazenados, quem pode consultar, que saídas são produzidas e que controles preservam o significado da informação.

Um sistema de informação pode apoiar uma operação diária, como registrar vendas; uma decisão, como analisar inadimplência; um controle, como verificar autorizações; ou uma prestação de contas, como produzir relatório para auditoria. Seu valor está na capacidade de tornar a informação disponível, compreensível, segura e rastreável.

É importante notar que um sistema de informação pode conter componentes computacionais e não computacionais. Uma regra de conferência, um procedimento de aprovação e uma política de acesso podem fazer parte do sistema tanto quanto uma tela ou banco de dados. Em organizações modernas, muitos sistemas de informação usam tecnologia digital, mas a lógica do sistema continua envolvendo pessoas, processos, dados e objetivos.

## Sistemas corporativos

Sistemas corporativos exigem mais rigor que aplicativos simples porque sustentam atividades com impacto operacional. Eles não apenas exibem informação; eles registram operações, preservam histórico, aplicam regras de negócio, controlam acesso, integram áreas, produzem evidências e precisam continuar funcionando de forma previsível.

O registro histórico permite saber o que ocorreu, quando ocorreu, quem executou, que dados foram usados e qual resultado foi produzido. Sem histórico, não há boa investigação, auditoria ou continuidade. Controle de acesso também é essencial, porque nem todo usuário pode consultar, alterar, aprovar ou excluir qualquer informação.

Consistência significa que as informações precisam fazer sentido entre si. Se um pagamento aparece como realizado em um sistema, mas como pendente em outro, a organização tem uma divergência. Rastreabilidade e auditoria permitem explicar operações depois que elas acontecem, especialmente em áreas em que decisões e registros precisam ser verificáveis.

Disponibilidade significa que o sistema precisa estar acessível quando a operação depende dele. Integração aparece porque sistemas corporativos raramente trabalham sozinhos: cadastro, cobrança, estoque, financeiro, contabilidade, atendimento e relatórios podem depender uns dos outros. Manutenção é necessária porque regras mudam, processos evoluem e erros são encontrados.

## Quando uma parte falha, o sistema sente

Em um sistema, uma falha localizada pode produzir efeitos amplos. Isso acontece porque as partes são relacionadas. Uma alteração pequena em um componente pode afetar dados, regras, interfaces, permissões, arquivos, integrações ou processos posteriores.

Imagine um cadastro de clientes em que o campo de documento passa a aceitar valores sem validação adequada. A princípio, a falha parece estar em um campo. Mas esse dado pode ser usado para emissão de nota, análise de crédito, cobrança, atendimento, auditoria e integração com parceiros. Um problema pequeno na entrada pode se espalhar por várias saídas.

Imagine uma rotina de cobrança que depende de um arquivo enviado por outro sistema. Se o arquivo chega com atraso, em formato diferente ou com registros incompletos, o sistema de cobrança pode gerar boletos incorretos, deixar cobranças de fora ou produzir relatórios divergentes. A falha não está apenas no programa que emite boletos; pode estar na interface entre sistemas.

## Conexão futura com Mainframe e COBOL

Mainframe e COBOL serão estudados futuramente como partes de sistemas corporativos maiores. Nesta fase, não é necessário aprender tecnicamente Mainframe, JCL ou COBOL. O ponto é compreender que essas tecnologias não existem isoladas. Elas participam de sistemas que processam dados, preservam histórico, executam regras, produzem saídas e sustentam operações críticas.

Um programa COBOL raramente deve ser imaginado como um exercício isolado. Em ambientes corporativos, ele pode participar de rotinas batch, processar arquivos, consultar bancos de dados, atualizar registros, compor fechamentos, gerar relatórios, validar regras de negócio ou integrar sistemas. Seu comportamento depende de entradas, saídas, ambiente, dados, arquivos, permissões, parâmetros, agendamento e operação.

O Mainframe, por sua vez, será estudado como ambiente de computação corporativa em que muitos sistemas precisam de confiabilidade, controle, escala, segurança e continuidade. Entender sistema ajuda o aluno a não olhar apenas para uma linguagem ou máquina. O foco passa a ser o conjunto: dados, programas, jobs, arquivos, bancos, usuários, controles, integrações e processos de negócio.

## Sistema como modelo de análise

“Sistema” é palavra usada em biologia, engenharia, computação e organizações. Nesta formação, sistema é conjunto de elementos e relações cujas interações produzem comportamentos, propriedades ou capacidades que não são compreendidos adequadamente ao olhar cada parte isoladamente. A definição é operacional, não universal.

Relações importam tanto quanto partes: fluxo, comunicação, dependência, restrição, coordenação e influência. Sistemas projetados geralmente têm finalidade atribuída por pessoas; sistemas naturais não precisam ter propósito intencional. Convém distinguir finalidade projetada, função desempenhada e comportamento observado.

Fronteira é delimitação analítica do que será considerado parte do sistema e pode mudar conforme a pergunta. O que fica fora, mas influencia ou é influenciado, compõe ambiente. Subsistema é parte analisável de sistema maior; interface é ponto de relação entre partes, não apenas tela. Entrada, transformação, saída e feedback são modelos úteis, não propriedades universais. Estado é condição relevante em certo momento; feedback é retorno que influencia comportamento posterior. Propriedades como disponibilidade, segurança e experiência de uso podem emergir da interação entre partes e ambiente.

## Confusões comuns de iniciantes

Uma confusão comum é achar que sistema é apenas uma tela. A tela é uma interface, isto é, um ponto de contato entre usuário e sistema. Ela permite entrada e visualização de saídas, mas por trás dela podem existir regras, dados, permissões, integrações, logs e processos. Reduzir o sistema à tela impede enxergar o funcionamento real.

Outra confusão é usar sistema como sinônimo de aplicativo ou de código. Um aplicativo pode ser apenas uma forma de acesso, e o código pode ser apenas uma parte do comportamento. O sistema maior pode depender de servidores, bancos de dados, autenticação, permissões, operação, documentação, suporte e manutenção. Por isso, um código tecnicamente correto ainda pode falhar dentro de um sistema mal integrado ou alimentado por dados ruins.

Também é comum acreditar que uma ferramenta sozinha resolve um problema. Uma ferramenta pode apoiar uma atividade, mas precisa estar inserida em um processo. Se uma empresa compra uma ferramenta de controle, mas não define dados obrigatórios, responsáveis, regras de uso e forma de acompanhamento, o problema continuará existindo.

Muitos iniciantes ignoram usuários, processos e dados. Isso leva à ideia de que basta "o sistema funcionar" tecnicamente. Porém, se o usuário não compreende o fluxo, se o processo exige etapas contraditórias ou se os dados são incompletos, inconsistentes ou mal definidos, o comportamento do sistema será afetado.

Também há dificuldade em perceber dependências entre partes. Um relatório pode depender de registros de várias áreas; uma cobrança pode depender de cadastro, contrato e pagamento; uma autorização pode depender de regra, permissão e histórico. Quando essas dependências não são vistas, mudanças parecem simples demais. Daí surge outra crença perigosa: imaginar que mudar uma parte não afeta o todo. Em sistemas, alterar um campo, uma regra, uma permissão, uma interface ou um horário de execução pode produzir efeitos em cadeia.

## Perguntas para reflexão

- Em que sentido uma pilha de peças ou documentos é diferente de um sistema?
- Qual é o propósito de um sistema de cadastro de clientes, além de guardar nomes?
- Que componentes podem existir em um sistema de cobrança além da tela usada pelo usuário?
- Como uma entrada incorreta pode afetar processamento, saída e decisões posteriores?
- Onde fica a fronteira de um sistema de atendimento ao cliente? O que fica dentro e o que fica fora?
- Que interfaces podem existir entre um sistema de vendas e outros sistemas de uma empresa?
- Por que uma falha em um campo de cadastro pode afetar relatório, cobrança ou auditoria?

## Síntese da aula

Sistema é um conjunto organizado de componentes relacionados que atuam com uma finalidade. Essa definição parece simples, mas muda a forma de observar tecnologia. Um sistema não é apenas um conjunto de partes, porque as relações entre as partes importam. Também não é apenas uma estrutura parada, porque sistemas recebem entradas, realizam transformações, produzem saídas e interagem com o ambiente.

Nesta aula, vimos que sistemas possuem componentes, relações, propósito, fronteira, ambiente externo e interfaces. Também vimos que podem ser naturais, sociais, técnicos, organizacionais, informacionais, computacionais ou corporativos. Esses tipos ajudam a perceber que sistemas computacionais fazem parte de contextos maiores.

A ideia principal é que sistemas precisam ser entendidos pelo comportamento do todo. Uma falha pequena pode ter grande impacto quando afeta dados, regras, interfaces, permissões, arquivos, integrações ou processos. Essa visão será usada em toda a formação.

A próxima aula tratará de hardware, software e pessoas. Agora que o aluno já entende que um sistema não é apenas uma parte isolada, será possível observar alguns componentes fundamentais dos sistemas computacionais e entender como eles participam do funcionamento conjunto.

## Mini glossário da aula

**Sistema**: conjunto organizado de componentes relacionados que operam para cumprir uma finalidade. Um sistema deve ser entendido pelo comportamento do conjunto, não apenas pelas partes isoladas.

**Componente**: parte identificável de um sistema que cumpre alguma função dentro do conjunto. Pode ser técnico, humano, documental, operacional ou organizacional.

**Relação**: vínculo ou interação entre componentes. As relações definem como as partes influenciam umas às outras e ajudam a explicar o comportamento do sistema.

**Propósito**: finalidade que orienta a existência do sistema. Sem propósito, não há critério claro para avaliar se o sistema funciona adequadamente.

**Fronteira**: limite conceitual que separa o que será considerado dentro do sistema e o que será considerado fora dele.

**Ambiente**: contexto externo ou conjunto de condições que influencia o sistema. O ambiente pode fornecer entradas, receber saídas ou impor restrições.

**Interface**: ponto de contato entre sistemas, componentes ou pessoas. Pode ser uma tela, arquivo, mensagem, formulário, canal de atendimento ou integração.

**Entrada**: dado, solicitação, documento, mensagem ou evento recebido pelo sistema para iniciar ou orientar uma operação.

**Processamento**: transformação realizada pelo sistema sobre entradas, de acordo com regras, etapas, decisões ou cálculos.

**Saída**: resultado produzido pelo sistema, como relatório, resposta, arquivo, atualização, registro, notificação ou decisão.

**Retroalimentação**: retorno de informações sobre resultados ou saídas para ajustar, corrigir ou melhorar o funcionamento do sistema.

**Sistema de informação**: sistema que coleta, organiza, processa, armazena, transmite e apresenta informações para apoiar operações, decisões, controles e prestação de contas.

**Sistema computacional**: sistema em que recursos computacionais participam diretamente do funcionamento, incluindo software, dados, hardware, usuários, permissões e ambiente.

**Sistema corporativo**: sistema usado para sustentar processos importantes de uma organização, com regras de negócio, histórico, controles, integrações e impacto operacional.

**Processo de negócio**: sequência organizada de atividades que uma organização executa para produzir um resultado relevante.

**Regra de negócio**: condição, cálculo, restrição ou decisão que expressa uma necessidade da organização e orienta o funcionamento do sistema.

**Integração**: comunicação ou troca de dados entre sistemas, componentes ou áreas, geralmente por meio de interfaces definidas.

**Dependência**: relação em que um componente, processo ou sistema precisa de outro recurso, dado, regra ou serviço para funcionar corretamente.

**Falha**: ocorrência em que um sistema, componente ou processo não produz o comportamento esperado.

**Operação**: execução organizada das atividades necessárias para que sistemas e processos funcionem no uso real.

## Critérios de domínio

Depois de estudar esta aula, o aluno deve conseguir explicar sistema com as próprias palavras, indicando que um sistema envolve componentes relacionados, organização e propósito. A explicação não deve se limitar à ideia de "conjunto de partes"; deve mostrar que relações e finalidade são essenciais.

O aluno deve conseguir diferenciar um conjunto qualquer de partes de um sistema. Deve ser capaz de dizer por que computadores comprados por uma empresa, documentos acumulados ou programas isolados não formam automaticamente um sistema funcional.

Também deve conseguir identificar componentes, relações e propósito em exemplos simples, como cadastro de clientes, abertura de conta, pagamento de salário, emissão de boleto ou atendimento ao cliente. O aluno deve perceber que processos de negócio podem ser analisados como sistemas.

O aluno deve explicar entrada, processamento, saída e retroalimentação em nível inicial, sem transformar esses conceitos em programação. A compreensão esperada é conceitual: sistemas recebem algo, transformam algo, produzem resultado e podem usar retornos para ajuste.

Também é esperado que o aluno diferencie sistema, software, aplicativo e ferramenta. Deve entender que software e aplicativos podem fazer parte de sistemas, mas não representam necessariamente o todo.

O aluno deve conseguir explicar por que sistemas corporativos envolvem mais que código. Deve mencionar dados, regras, processos, usuários, controles, integração, histórico, auditoria, disponibilidade, manutenção e impacto operacional.

Por fim, deve conectar o conceito de sistema com o futuro estudo de Mainframe e COBOL, compreendendo que programas, arquivos, bancos, jobs, relatórios e rotinas corporativas serão partes de sistemas maiores, e não elementos isolados.
